#+title:  GDPR Excision with Per-Tenant Databases
#+author: James Conroy-Finn
#+date:   2026-02-15

* Overview
Implement GDPR Article 17 ("right to erasure") using per-tenant Datomic
databases. Each creator gets their own database. Creator deletion = drop
database. User (subscriber) deletion = excision within tenant's database.

* Architectural Decision
Per-tenant Datomic databases (WordPress model):

- Each creator/tenant gets their own Datomic database
- Users are tenant-scoped (same email can exist in multiple tenants)
- Creator says "delete my data" → drop their database
- User (subscriber) says "delete my data" → excision within that tenant's database
- Clean isolation, simpler GDPR compliance

* Connection Management
** Connection Pool
Maintain an in-memory map of tenant-id → Datomic connection. When serving a
tenant:

1. Check if connection exists in pool
2. If not, establish connection and cache it
3. Put connection into request map as =:datomic=

Downstream code sees =:datomic= - no "platform connection" or "tenant
connection" distinctions. The middleware handles resolution transparently.

#+begin_src clojure
;; Conceptual - connection resolution is transparent to handlers
(defn wrap-datomic
  [handler]
  (fn [request]
    (let [tenant-id (get-in request [:session/realm :tenant/id])
          conn      (get-or-create-connection! tenant-id)]
      (handler (assoc request :datomic conn)))))
#+end_src

** Pool Lifecycle
- Connections established lazily on first request to tenant
- Connections closed when tenant is deleted
- Consider connection health checks and reconnection

* Creator Deletion
Creator requests account deletion → delete their entire tenant.

** Deletion Request Schema (Platform DB)
#+begin_src clojure
:tenant-deletion/id           UUID
:tenant-deletion/tenant       Ref
:tenant-deletion/status       Keyword (:pending, :in-progress, :completed, :cancelled)
:tenant-deletion/requested-at Instant
:tenant-deletion/scheduled-at Instant (configurable cooling-off)
:tenant-deletion/completed-at Instant
#+end_src

** Processing
1. Mark status =:in-progress=
2. Remove connection from pool
3. Delete the Datomic database (=d/delete-database=)
4. Clean up PostgreSQL (sessions, auth attempts for this tenant)
5. Mark status =:completed=

No excision needed - entire database is gone.

** Subscriber Notification

** Excision Plan
Build transaction to excise user and related entities:

#+begin_src clojure
(defn excision-plan
  [db user-eid]
  (let [memberships (d/q '[:find [?m ...] :in $ ?u
                           :where [?m :membership/user ?u]]
                         db user-eid)]
    (mapv #(hash-map :db/excise %)
          (into [user-eid] memberships))))
#+end_src

** Processing
1. Delete PostgreSQL data (sessions, auth attempts)
2. Execute excision transaction
3. Wait for =d/sync-excise=
4. User can re-register with same email (clean slate)

** Processing Loop
#+begin_src clojure
(defn reap!
  [reaper]
  ;; Process tenant deletions (platform level)
  (process-tenant-deletions! reaper)

  ;; Process user deletions (per-tenant)
  (doseq [tenant (active-tenants reaper)]
    (process-user-deletions! reaper tenant)))
#+end_src

** Cooling-Off Adjustment
Support staff can adjust =scheduled-at= via email request to delay or expedite
deletion.

* PostgreSQL Cleanup
Sessions and auth attempts are tenant-scoped:

#+begin_src clojure
(defn delete-user-sessions!
  [postgres tenant-id user-id]
  (postgres/execute! postgres
    {:delete-from :sessions
     :where [:and
             [:= :tenant-id tenant-id]
             [:= :user-id user-id]]}))

(defn delete-tenant-sessions!
  [postgres tenant-id]
  (postgres/execute! postgres
    {:delete-from :sessions
     :where [:= :tenant-id tenant-id]}))
#+end_src
